# Backend API documentation migration: Swagger file -> zod-openapi (OpenAPI 3.1)

Date: 2025-11-06
Base branch: `dev`

This document summarizes what changed, where and how; it compares the old implementation with the new one, and lists pros and cons.

## Overview

The backend switched from a generated Swagger JSON file (`swagger-output.json` via `swagger-autogen` and `src/swagger.ts`) to a type-safe, programmatic OpenAPI 3.1 document built with `zod` + `zod-openapi`.

Key themes:
- Single source of truth: request/response schemas defined with Zod are reused for runtime validation and for OpenAPI generation.
- No more build-time swagger generation/watchers.
- API docs are served directly from code as `/api-docs` (UI) and `/api-docs.json` (JSON).

## What changed (by file)

- apps/backend/package.json
  - Removed scripts: `dev:full`, `build:swagger`, `build:swagger:clean`, `build:swagger:copy`, `swagger:watch`.
  - Removed dependency: `swagger-autogen`.
  - Added dependency: `zod-openapi`. Bumped `zod` to `^4.1.12`.
- apps/backend/src/server.ts
  - Old: loaded a `swagger-output.json` file from `src` or `dist` with `fs/path`, then served it via Swagger UI and `/api-docs.json`.
  - New: imports `openApiDocument` from `./api/openapi/document` and passes it to Swagger UI and `/api-docs.json`.
- apps/backend/src/api/controllers/auth/AuthController.ts
  - Old: defined inline Zod schemas for auth flows inside the controller.
  - New: imports shared schemas from `../../middleware/validation` to centralize validation and documentation.
- apps/backend/src/api/middleware/validation.ts
  - Expanded and centralized validation schemas. Added `.meta(...)` and `.meta({ id: ... })` to enrich OpenAPI output.
  - New schemas added (e.g., `segmentQuerySchema`, `updatesQuerySchema`).
- Removed files
  - apps/backend/src/swagger.ts
  - apps/backend/src/swagger-output.json (generated file no longer needed)
- New files (untracked before this operation)
  - apps/backend/src/api/openapi/document.ts — base OpenAPI document and `createDocument` call.
  - apps/backend/src/api/openapi/routes.ts — endpoint path definitions tied to Zod schemas.
  - apps/backend/src/api/openapi/schemas.ts — common response wrapper schemas.
- package-lock.json
  - Updated to reflect dependency changes.

## Old vs New implementation

### Old (Swagger autogen + generated JSON)
- Location of spec: `src/swagger-output.json` (copied to `dist/` in builds) generated by `src/swagger.ts` and build scripts.
- Server integration: `server.ts` read JSON from file system and served it.
- Validation: Zod used in places, but schemas were not consistently tied to the Swagger generation.
- Dev experience: separate watcher or build step to re-generate the swagger JSON.

### New (zod-openapi + code-first OpenAPI 3.1)
- Location of spec: constructed at runtime via `openApiDocument` from `apps/backend/src/api/openapi/document.ts`.
- Server integration: `server.ts` imports `openApiDocument`; UI at `/api-docs`, JSON at `/api-docs.json`.
- Validation: Zod schemas in `middleware/validation.ts` are the single source for request/response validation and the OpenAPI schema generation (with `.meta` annotations for examples, descriptions, IDs).
- Dev experience: no generation step; change Zod schemas or routes and the docs update immediately on restart.

## Notable code changes (excerpts)

- server.ts
  - Replaced file I/O based swagger loading with `import { openApiDocument } from './api/openapi/document'` and piped that to Swagger UI and `GET /api-docs.json`.
- validation.ts
  - Added comprehensive Zod schemas with `.meta` metadata and IDs for OpenAPI. Centralizes previous inline schemas from controllers.
- package.json (backend)
  - Removed swagger-related scripts and `swagger-autogen`. Added `zod-openapi` and updated `zod`.

## Pros and cons

### Pros
- Single source of truth: validation and documentation share Zod schemas, reducing drift.
- Type-safety and maintainability: schemas live in code, easier to refactor and review.
- No generated file: eliminates watcher/build complexity and large JSON diffs.
- Better docs: examples, descriptions, and IDs via Zod `.meta` improve Swagger UI.
- OpenAPI 3.1 ready: modern spec with improved schema capabilities.
- Faster iteration: change code, restart server, docs update immediately.

### Cons / considerations
- Learning curve: contributors need familiarity with `zod-openapi` patterns.
- Runtime generation: spec is built at startup/import time (usually negligible cost, but not zero).
- Removed scripts: any CI tasks or docs referencing `build:swagger` or the generated JSON file must be updated.
- Coverage discipline: endpoints must be declared in `openapi/routes.ts`; easy to forget a new route until a test/check enforces it.

## How to view the docs now
- Swagger UI: GET `http://localhost:<PORT>/api-docs`
- Raw JSON: GET `http://localhost:<PORT>/api-docs.json`

## Follow-ups (optional)
- Add a small CI check that ensures `openApiDocument` builds without warnings.
- If a static spec artifact is still desired for consumers, add a task to write `openApiDocument` to a JSON file during the build (e.g., `dist/openapi.json`).
- Lint rule or test to ensure every new Zod schema includes `.meta({ id: ... })` and useful examples.
